## カテゴリ C — 音声ファイル紐づけの精度・運用問題

| # | 問題 | 重要度 | 対応状況 |
|---|------|--------|---------|
| C-1 | ファイル名正規表現が非貪欲マッチのため、担当者名に `_` が含まれると最初の `_` で切れる（例: `20260424_営業部_山田太郎_001.mp3` → `"営業部"` になる） | 🔴 高 | ✅ 対応済み |
| C-2 | タイムスタンプ許容誤差 ±10分は展示会環境では広すぎる。5〜10分で来場者が入れ替わるブースで同一担当者の連続対応ケースに誤紐づけが発生しやすい | 🔴 高 | ✅ 対応済み |
| C-3 | 複数ファイルが同一リードに紐づく場合「録音時間が長い方を採用」しているが根拠がない。また mutagen 未インストール時は全ファイルが `duration_sec=0.0` になり判定が機能しない | 🔴 高 | ✅ 対応済み |
| C-4 | 担当者名不明（red）のファイルが多数あると一括対応できず、個別セレクトボックスを全件スクロールする必要がある。スキップ方法もUIから分かりにくい | 🟡 中 | ✅ 対応済み |
| C-5 | ニーズ抽出に渡す文字起こしを4000文字で切り捨てており、5〜10分の会話で後半の予算・決裁者情報が失われる可能性がある | 🟡 中 | ✅ 対応済み |
| C-6 | ID3タグの録音日時解析が `datetime.fromisoformat()` のみ対応で、タイムゾーン付きや非ISO形式のフォーマットで `start_time = None` になりタイムスタンプマッチが使えなくなる | 🟢 低 | ✅ 対応済み |
| C-7 | MP3の TDRC タグしか参照しておらず、iPhone（m4a/`©day`）や古い録音機器（TYER+TDAT）など非MP3フォーマットの録音日時が常に `None` になる | 🔴 高 | ✅ 対応済み |

### C カテゴリ 対応詳細

#### C-1: ファイル名正規表現を貪欲マッチに変更
- **対象ファイル**: `src/audio_matcher.py`
- **変更内容**: `_FILENAME_PATTERN` と `_CSV_FILENAME_PATTERN` の担当者名キャプチャ部分を `(.+?)` (非貪欲) → `(.*)` (貪欲) に変更
- **効果**: 最後の `_連番` の直前まで全部を担当者名として取得するようになり、`20260424_営業部_山田太郎_001.mp3` → `"営業部_山田太郎"` と正しく解析される

#### C-2: タイムスタンプ許容誤差を5分に変更
- **対象ファイル**: `src/config.py`
- **変更内容**: `AUDIO_TIMESTAMP_TOLERANCE_MINUTES: 10 → 5`
- **効果**: 5分以内を同一来場者との会話として判定することで、連続来場者への誤紐づけリスクを低減

#### C-3: 複数ファイル重複時の選択基準を「時刻差が近い方」に変更
- **対象ファイル**: `src/audio_matcher.py`
- **変更内容**: `timestamp_matched` の格納値を `lead_idx` から `(lead_idx, timedelta)` に変更し、重複時は録音時間ではなくQRスキャン時刻との差分が小さい方を採用するロジックに変更。紐づけ結果の note に時刻差（秒）も表示するよう改善。
- **効果**: mutagen 未インストール環境でも正しく動作し、タイムスタンプマッチングの目的と一貫した選択基準になる

#### C-4: red ファイルのUI改善
- **対象ファイル**: `app.py`
- **変更内容**: 🟢🟡の自動紐づけ結果と🔴を分離し、redファイルをエクスパンダーにまとめて件数をタイトルに表示。「（紐づけなし）のまま確定を押せばスキップされる」旨の説明文を追加。確定ボタン横にスキップ件数の案内キャプションを追加。
- **効果**: red が多数ある場合にスクロール負荷が減り、スキップ手順がUIから分かるようになる

#### C-5: 文字起こし切り捨て上限を拡大
- **対象ファイル**: `src/audio_processor.py`
- **変更内容**: `transcript[:4000]` → `transcript[:8000]`
- **効果**: 5〜10分の展示会会話で後半に出てくる予算・決裁者・具体的なニーズがニーズ抽出LLMに渡るようになる

#### C-6: ID3タグのタイムスタンプ解析を堅牢化
- **対象ファイル**: `src/audio_processor.py`
- **変更内容**: `datetime.fromisoformat(raw[:19])` → `pd.to_datetime(raw)` + `.replace(tzinfo=None)` に変更。`import pandas as pd` も追加。
- **効果**: タイムゾーン付き（例: `2026-04-24T14:30:00+09:00`）や非ISO形式など、録音機器が出力する多様なフォーマットに対応し、`start_time = None` による信頼度低下を防ぐ

#### C-7: 複数音声フォーマットの録音日時タグに対応
- **対象ファイル**: `src/audio_processor.py`
- **変更内容**: タイムスタンプ抽出をモジュールレベルの `_extract_recording_time()` 関数に切り出し、以下の順で複数タグを試行するように変更

  | 優先順 | タグ | 対応フォーマット・機器 |
  |--------|------|----------------------|
  | 1 | `TDRC` | MP3/ID3v2.4（Android録音アプリ、ICレコーダー等） |
  | 2 | `©day` | M4A/MP4 iTunes形式（**iPhone Voice Memo 等**） |
  | 3 | `TYER`+`TDAT`+`TIME` | MP3/ID3v2.3（古いファームウェアの録音機器） |
  | 4 | `TDRL` | ID3v2.4（TDRCが存在しない機器の代替タグ） |

- **効果**: iPhoneで録音した m4a ファイルでも録音日時が取得できるようになり、タイムスタンプマッチング・sequential ソートの精度が向上する

#### 順番マッチング信頼性向上（C-1〜C-7 に続く追加対応）
- **対象ファイル**: `src/audio_matcher.py`, `app.py`
- **変更内容**:
  1. `app.py`: `matcher.match()` に `rep_col` / `timestamp_col` / `tolerance_minutes` を渡すよう修正。`timestamp_col` は新設の `detect_timestamp_col()` で自動検出（標準列名 → キーワード → LLM の3段階）
  2. `audio_matcher.py`: sequential マッチング時もリードを `scan_time` 昇順にソート（従来は DataFrame 行順のまま）
  3. `audio_matcher.py`: 音声ファイルのソートに `start_time` 優先・なければファイル名末尾の連番を使用（従来は全て `datetime.min` 扱いでアップロード順依存）
  4. `audio_matcher.py`: `detect_timestamp_col()` メソッドを追加。`scan_time` 列が存在しない CSV でも datetime 解析可能な列を自動検出し、複数候補は LLM で判定
- **効果**: LeadManager の「scan_time 順の記録」と「音声連番順」の対応が正しく取れるようになり、後日 CSV を提出しなくても担当者名一致 + 順番で高精度な紐づけが成立する
