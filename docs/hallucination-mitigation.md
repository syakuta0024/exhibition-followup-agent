LLM アプリケーションを開発していると、必ずぶつかる問題がある。「プロンプトに書いていない
ことが出力に出てくる」——いわゆるハルシネーションだ。

このドキュメントは、Phase 3.3.5 で実際に遭遇した2件のハルシネーション事案を記録する。
仮説が外れた過程、真因の特定、対策の設計判断を残すことで、同種バグへの再現性のある対処
フローを作ることが目的だ。

## 2. 何が起きたか

### 2.1 事象1：lead_id 漏洩（L007 問題）

**症状**

メール生成後の出力確認で、件名や本文に「L007」「L001」のような文字列が混入していた。
これは内部管理上のリード ID であり、来場者に送るメールに記載されるべき情報ではない。

**なぜ問題か**

- 顧客が「L007」という文字列の意味を理解できない。不審なメールとして処理される
- 社内の管理番号体系が外部に露出する（意図しない情報開示）
- ビジネスメールとしての信頼性・プロとしての印象を損ねる

### 2.2 事象2：ダミー URL 混入

**症状**

複数リードへの生成メールに、実在しない URL が混入していた。製品紹介の文脈で
「詳細はこちら: `https://example.com/edgeguard`」のように、LLM が自己生成した URL が
本文に含まれる形だった。

**なぜ問題か**

- リンク切れ・架空 URL を顧客に送信することは信頼を著しく損ねる
- 製品 URL を設定していないにもかかわらず、LLM が「ビジネスメールには参照 URL が必要」
  と推論して架空 URL を補完していた

## 3. 仮説 → 検証 → 方針転換のプロセス

### 3.1 当初の仮説

**仮説1（lead_id 漏洩）**

> 「プロンプトのどこかに lead_id が含まれており、LLM がそのまま出力している」

FollowUpAgent がリード情報をプロンプトに展開する際、意図せず lead_id フィールドも
含めているのではないか、という仮説。

**仮説2（ダミー URL）**

> 「`data/tech_documents/` 内の Markdown ファイルにデモ用 URL が書かれており、RAG 経由
> で取得されてメールに混入している」

ナレッジベース構築時に投入した技術資料に、テスト用の URL が含まれていた可能性を疑った。

どちらの仮説も「プロンプトを調べれば解決できる」という前提に立っていた。

### 3.2 検証で判明した事実

**検証方法**

プロンプト組み立て後の全文を出力し、問題の文字列が含まれているかを目視および
文字列検索で確認した。

**結果**

| 仮説 | 検証結果 |
|---|---|
| プロンプトに lead_id が含まれる | **含まれていない**（コミット `fbc4e8e` のメッセージに記録） |
| tech_documents にダミー URL が含まれる | **含まれていない** |

両仮説が外れた。問題の文字列はプロンプトのどこにも存在しなかった。

### 3.3 真因の特定

「プロンプトにない → 出力にある」——これはハルシネーションの典型的な構造だ。

**真因**

LLM が学習データから以下の推論パターンを持ち込み、プロンプトに存在しない情報を
自己生成していた。

- 「ビジネスフォローアップメールには参照 ID が必要だ」→ L007 を生成
- 「製品紹介を含むメールには資料 URL の案内が必要だ」→ example.com 等を生成

「そう書くのがビジネスメールとして自然だ」という学習パターンに従い、実際のプロンプト
内容を超えて出力を補完した。プロンプトがどれだけ整っていても、学習データに由来する
バイアスは排除できない。

## 4. 対策（What & How）

### 4.1 明示的な禁止指示の追加

**`_build_output_format_instruction()` への追加**
（[src/email_generator.py:141](../src/email_generator.py#L141)）

```
※ 件名・本文に「L001」「L007」のような内部管理コード・参照IDを含めないでください。
```

**`_build_system_prompt()` 末尾への追加——二重防衛**
（[src/email_generator.py:271](../src/email_generator.py#L271)）

```
件名・本文に「L001」「L007」のような内部管理IDを含めないでください。
```

同じ禁止指示をシステムプロンプトとユーザープロンプトの2箇所に配置した。LLM は同じ
制約を複数箇所で確認することで、その制約をより強く守る傾向がある。「二重防衛」と呼ぶ。

### 4.2 `cli_config.yaml` の `product_urls`——「器」の設計

コミット `0596d25` で `cli_config.yaml` に `product_urls` セクションを追加した。

```yaml
# cli_config.yaml（現在の状態）
product_urls:
  DigiMA: ''
  EdgeGuard: ''
  FactoryBrain: ''
  NTX-OCR: ''
  SmartVision: ''
  Sorani: ''
```

初期値は全て空文字列——値の「器」だけ作り、URL は未設定の状態としてリリースした。

**設計の核心**：URL の有無でプロンプトの禁止指示を切り替える。

- **値が空文字列（未設定）** → URL 全面禁止を明示
- **値が設定されている** → 設定済み URL のみ許可、それ以外は明示的に禁止

「URL を設定しない限り架空 URL を生成させない」という安全側への倒し方だ。

### 4.3 伝播チェーン

`product_urls` は設定ファイルから読み取られ、メール生成プロンプトまで一貫して伝播する。

```
cli_config.yaml
  └─ cli_runner.run_generate()
       _product_urls = config.get("product_urls", {})   ← src/cli_runner.py:356
           │
           └─ agent.FollowUpAgent.process_lead()         ← src/agent.py:59, 229
                product_urls=product_urls
                    │
                    └─ email_generator.EmailGenerator.generate()   ← src/email_generator.py:187, 220
                         product_urls=product_urls
                             │
                             └─ _build_output_format_instruction(
                                    sender_company, sender_name, product_urls
                                )                        ← src/email_generator.py:393
```

### 4.4 結果検証

L001 で動作確認を実施した。確認した観点は3点。

- 内部 ID（L001 等）が件名・本文に含まれないこと
- URL が含まれないこと（product_urls が全て空文字列の状態）
- メール品質（ランク別トーン・構成）が維持されること

3点ともパスを確認した。

## 5. 学び：「変数展開漏れ」と「ハルシネーション」の見分け方

LLM アプリのバグは大別すると2種類ある。見た目が似ているが原因も対処も異なる。

### 5.1 変数展開漏れ（Phase 3.2.5 の事例）

**症状**：プロンプトのリテラル文字列（例: `{会社名}`）がそのまま出力に混入する。

**発生構造**：f-string や `.format()` の変数展開が不完全で、プレースホルダーが
未解決のまま LLM に渡される。LLM はそのリテラルを「出力すべき文字列」と解釈する。

**検証**：プロンプト変数展開後の全文を出力 → 該当文字列が「含まれている」

**対処**：展開ロジックのバグ修正（Python 側の問題）

### 5.2 ハルシネーション（Phase 3.3.5 の事例）

**症状**：プロンプトに「ない」ものが出力に出る（L007、example.com 等）。

**発生構造**：LLM が学習データのパターンから「こう書くのが自然だ」と判断し、
プロンプトを超えて情報を補完する。

**検証**：プロンプト変数展開後の全文を出力 → 該当文字列が「含まれていない」

**対処**：プロンプトへの明示的な禁止指示の追加（§4 参照）

### 5.3 見分けフローチャート

```
出力に不審な文字列を発見
         │
         ▼
プロンプト全文（変数展開後）を出力する
         │
    ─────┴──────────────────────────────
    │                                  │
    ▼ 含まれている                      ▼ 含まれていない
変数展開漏れ系                      ハルシネーション系
    │                                  │
Python 側の展開ロジックを修正     明示的禁止指示をプロンプトに追加
```

「プロンプト全文出力 → 目視確認」は LLM アプリのデバッグで最初にすべき手順だ。
これをしないと、原因がコードにあるのかモデルにあるのかの切り分けができない。

## 6. 禁止指示の効果と限界

### 6.1 効果

- L001 での動作確認で有効性を確認（内部 ID・ダミー URL の消滅）
- 実装コストがほぼゼロ（プロンプト文字列の追加のみ）
- 即時適用可能（コード変更のみ、外部サービス設定不要）

### 6.2 限界

**100% の防止は保証されない**。LLM はブラックボックスであり、稀に明示的な禁止指示を
無視することがある。特に以下の条件下でリスクが高まる。

- 禁止指示が多くなりすぎてプロンプトが肥大化し、LLM の注意が分散する
- 指示と矛盾する強いコンテキストが他の箇所にある
- モデルバージョンアップで挙動が変わる

禁止指示は「事前の予防」であり、「生成結果の事後検証」にはならない。

### 6.3 将来フェーズで計画する補完策

現時点の「禁止指示のみ」は MVP として有効だが、スケールするにつれ不十分になる。
3層検証アーキテクチャを計画している。

```
Layer 1: Python による機械チェック
  ├─ 出力テキストに特定パターン（L\d+、example\.com 等）を regex で検査
  └─ 違反があればリトライまたはフラグ付きで保留

Layer 2: LLM-as-a-Judge
  └─ 別 LLM 呼び出しで「このメールはビジネスとして適切か」を自動評価

Layer 3: 人間確認（Gmail 下書き）← 現在ここだけ稼働
  └─ Gmail 下書きに保存し、担当者が送信前に内容を確認・修正
```

現在は Layer 3（人間確認）のみが機能している。Layer 1・2 は将来の実装課題。

## 7. まとめ

Phase 3.3.5 で遭遇したハルシネーション問題を振り返ると、核心は3点だ。

**仮説検証が方針転換の起点だった**

「プロンプトを出力して文字列を検索する」という単純な手順が、仮説の棄却につながった。
「プロンプトに存在しない」という事実を確認できたことで、初めてハルシネーションとして
問題を再定義できた。仮説を持ち、検証し、外れたら捨てる——このサイクルを省略すると
的外れな対処に時間を費やす。

**MVP 段階では「明示的禁止」が最速の現実解**

禁止指示の追加はコストゼロ・即時適用可能だ。100% の保証はないが、L001 で確認した
限りでは十分に有効だった。「完璧なソリューション」より「今動く対策」を優先するのが
MVP での正しい判断だと考えている。

**「余地を減らす」アーキテクチャが根本的アプローチ**

禁止指示は「LLM に正しく振る舞わせる」試みだが、限界がある。より根本的なのは、
LLM が誤れる余地をシステム側で減らすことだ。`product_urls` の「設定時だけ URL を
許可、未設定時は完全禁止」という設計はその思想の小さな実装だ。将来の Python 機械
チェック層もこの延長にある。
