# 将来のRAG拡張設計メモ

## 背景

本システム（v2）の現状RAGはMarkdown + 親子チャンク + ハイブリッド検索（ChromaDB + BM25 + RRF）で構成されている。これを次フェーズで以下のように拡張することを検討中。

## 設計思想

製造業の技術資料は図表が多く、単純なテキスト抽出では情報が失われる。VLMによる文字起こし、構造化メタデータ、Reranking、Contextual Retrieval を組み合わせることで、RAG精度を飛躍的に向上させる。

## 拡張パイプライン

```
PDF
  ↓ ① VLM文字起こし（図表を含めて完全テキスト化）
テキスト
  ↓ ② 構造化（製品/機能/事例の階層 + メタデータ付与）
構造化データ
  ↓ ③ 親子チャンキング（構造化単位でチャンク分割）
チャンク群
  ↓ ④ Contextual Retrieval（各チャンクに文脈プレフィックス付与）
コンテキスト付きチャンク
  ↓ ⑤ ベクトルDB格納（メタデータ付き）

[検索時]
クエリ
  ↓ ⑥ ハイブリッド検索（ベクトル + BM25 + RRF）← v2で実装済み
Top 100候補
  ↓ ⑦ Reranking（Cross-Encoder で再スコアリング）
Top 5件
  ↓
LLMへ渡す
```

## 各ステップの設計

### ① VLM文字起こし

- 候補: Claude / GPT-4o / Gemini
- 図やグラフは説明文を生成
- 表は構造化テキストに変換

### ② 構造化スキーマ案

```yaml
製品名: "EdgeGuard"
カテゴリ: "異常検知"
対象業界: ["製造業", "プラントエンジニアリング"]
主要機能:
  - 機能名: "振動センサー異常検知"
    詳細: "..."
適用ユースケース:
  - シナリオ: "プレス機の異常検知"
    効果: "月間47件の異常を事前検知"
技術スペック:
  処理速度: "リアルタイム（<100ms）"
```

### ③ 親子チャンキング

- 親チャンク: 製品全体概要（コンテキスト用）
- 子チャンク: 機能・事例・スペック（検索用）

### ④ Contextual Retrieval

- 各チャンクに「これは○○製品の××セクション」という文脈プレフィックスを付与
- Anthropic 2024 提唱手法、検索精度向上が報告されている

### ⑤ メタデータ

- 製品名、カテゴリ、業界、ユースケース等で検索時のフィルタリングが可能になる

### ⑥ ハイブリッド検索（実装済み）

- v2 で実装済み

### ⑦ Reranking

- 候補: Cohere Rerank API / BGE-Reranker / Cross-Encoder
- ベクトル検索で広く取って、Reranker で絞る

## ポートフォリオ文脈での意義

- 製造業の技術資料は図表が多く、単純なRAGでは限界がある
- メタデータ・Reranking・Contextual Retrieval は2026年時点のRAGベストプラクティス
- 製造業DX × AIエージェント × 高精度RAG という差別化ポイントになる

## 実装の優先順位（暫定）

1. Contextual Retrieval（既存コードへの追加が最も簡単）
2. メタデータ拡充（構造化スキーマの設計）
3. Reranking 導入
4. VLM文字起こしパイプライン
5. 構造化フローの自動化
