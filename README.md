# 展示会フォローアップ AI エージェント

展示会で集めた数百件のリードに対し、一人ひとりの興味と背景に合わせた個別最適化メールを生成し、
Gmail 下書きに保存する AI システム。リード 18 件をクリーン環境から 8 分で処理完了。

---

## なぜ作ったか

展示会後のフォローアップは「数百件 × 個別文面」という現実的な課題がある。

- **テンプレート一斉送信** → 個別性が出ず、興味のない製品の宣伝で逆に印象が悪化する
- **ChatGPT に都度貼り付け** → 非効率で、生成内容の検証も困難
- **ハルシネーション問題** → 架空の URL、関心外製品の言及、事実誤認はビジネス文書として致命的

「RAG × 音声録音 × Web 検索」を統合してこの課題を解決しようとした。

---

## 何を作ったか

リード CSV を読み込み、製品資料 RAG・音声録音紐づけ・Web 検索を統合して個別最適化メールを生成、
Gmail 下書きに一括保存するシステム。

操作は Claude Code の対話型 Skill (`/setup` → `/email-workflow`) のみ。
初見ユーザーでも 8 分でメール生成まで到達できる。

```
/setup          # 初回セットアップガイド（9ステップ）
/email-workflow # メール生成 → Gmail 下書き保存
```

## 技術的な工夫

### (a) ハイブリッド検索 + 親子チャンク

製品資料 RAG でベクトル検索(ChromaDB)と BM25 を RRF(Reciprocal Rank Fusion)で統合した。

**問題意識**: ベクトル検索は類似語に強いが固有名詞に弱い。BM25 は逆。どちらか一方では抜け漏れが生じる。

| 検索手法 | 強み | 弱み |
|---|---|---|
| ベクトル検索 (ChromaDB) | 意味的類似度 | 固有名詞の完全一致が弱い |
| BM25 | 固有名詞・専門用語 | 表記ゆれに弱い |
| RRF 統合 | 両方の長所 | — |

また親子チャンク設計(子=検索精度・親=文脈保持)を採用した。  
結果: 「Sorani」のような固有名詞でも確実にヒットするようになった。

### (b) 3層検証アーキテクチャ（ハルシネーション対策）

開発過程で **2 種類のハルシネーション**を発見・分類した:

1. **変数展開漏れ** — `{product_name}` 等のプレースホルダがそのまま出力される
2. **LLM 自己生成** — 存在しない URL や、リードが興味を示していない製品への言及

それぞれを別レイヤーで対処する設計に到達した:

| Layer | 手法 | 検出対象 |
|---|---|---|
| Layer 1 | Python 機械チェック（4 種のルール） | 内部 ID 漏れ・example.com・未設定 URL・関心外製品の言及 |
| Layer 2 | LLM-as-a-Judge（任意有効化、コスト管理可能） | 論理矛盾・文脈逸脱 |
| Layer 3 | Gmail 下書きで送信前の人間確認 | 最終確認 |

「ハルシネーション率 N% 削減」のような比較は測定していないが、
**検出できる種類が明確に定義されたこと**が品質改善の基盤になった。

### (c) Claude Code Skills 活用（v1 → v2 ピボット）

**v1**: Streamlit ベース UI を実装した。  
→ UI の保守・改善コストが高く、本質機能(RAG 精度・ハルシネーション対策)に時間が割けなくなった。

**v2(現在)**: Claude Code Skills(Markdown による対話定義)に全面ピボット。  
→ UI コードをすべて削除し、操作フローは Markdown 1 ファイルで定義。

**判断**: 動くものを捨てるのは心理的コストが高かったが、  
「ユーザーインターフェースは Claude Code が担う」と割り切ったことで保守性が大幅に向上した。

## 結果と学び

| 項目 | 測定値 |
|---|---|
| 自動テスト | 117 件パス（Phase 2 の 52 件 → Phase 6 で 117 件に拡充） |
| 実 LeadManager CSV カラム自動マッピング | 13 / 15 = 87% |
| エンドツーエンド処理（リード 18 件） | 8 分・エラー 0 件（Phase 8 実測） |
| 製品 RAG ヒット率 | 18 件中 18 件で関連資料 2〜3 件を取得 |
| 必須フィールドマッピング成功率 | Phase 2.5 で 10/11 → Phase 6 で 11/11 |

**学んだこと**:

- **MVP 順序の重要性** — 「動くもの → 品質改善 → 検証強化」の順で進めたことで、本当に必要な機能が見えてきた
- **ハルシネーション対策は「層」で考える** — 単一の対策より、検出 × 訂正 × 人間確認の多層構造が堅牢
- **データ駆動設計** — コード分岐より「ランク別プロンプトテンプレート」の方が保守性が高い(Phase 3.3.6 で実証)
- **KB 破壊バグの経験** — ナレッジベース再構築時に既存チャンクが消えるバグ(Atomic 原則違反)を発見・修正。副作用のない設計の重要性を体感した

| 分類 | 技術 |

## セットアップ

```bash
# 1. 依存パッケージをインストール
pip install -r requirements.txt

# 2. .env を作成して OPENAI_API_KEY を設定
echo "OPENAI_API_KEY=sk-..." > .env

# 3. Claude Code で /setup を実行
/setup
```

### 主な Skill

| Skill | 機能 |
|---|---|
| `/setup` | 初回セットアップガイド（9 ステップ） |
| `/email-workflow` | メール生成 → Gmail 下書き保存 |
| `/inspect-data` | リード CSV データ品質チェック |
