# 展示会フォローアップ AI エージェント

展示会で集めたリード情報から、一人ひとりの興味・業種・商談状況に合わせた個別最適化メールを AI が生成し、Gmail 下書きに一括保存するシステム。

---

## 背景と課題

展示会後のフォローアップには「数百件 × 個別文面」という実務的な壁がある。

- **テンプレート一斉送信** — 個別性が出ず、興味のない製品の紹介で印象が悪化する
- **ChatGPT への手貼り付け** — 非効率で、生成内容の検証も困難
- **ハルシネーション** — 架空の URL、関心外製品への言及、事実誤認はビジネス文書として致命的

---

## 概要

以下の情報を統合し、リードごとに最適化されたフォローアップメールを生成する。

- **リード情報**（展示会 CSV）: 氏名・会社名・興味製品・コメント・商談ランク
- **製品技術資料**（RAG）: Markdown 形式の製品ドキュメントから関連情報を検索・引用
- **CRM 情報**: 過去商談記録との照合でパーソナライズ精度を向上
- **音声録音**: 展示ブースでの会話を文字起こしし、メール素材として活用
- **Web 検索**: 訪問企業の最新ニュース・製品情報を動的に取得

生成したメールは **3 層で品質検証**したうえで Gmail 下書きに保存。  
操作は Claude Code の対話型 Skill（`/setup` → `/email-workflow`）のみ。

```
/setup          # 初回セットアップガイド（9 ステップ）
/email-workflow # メール生成 → Gmail 下書き保存
```

---

## 技術的な工夫

### (a) ハイブリッド検索 + 親子チャンク

製品資料 RAG にベクトル検索（ChromaDB）と BM25 を RRF（Reciprocal Rank Fusion）で統合した。

**問題意識**: ベクトル検索は意味的な類似度に強いが固有名詞に弱い。BM25 はキーワード一致に強いが表記ゆれに弱い。どちらか一方では抜け漏れが生じる。

| 検索手法 | 強み | 弱み |
|---|---|---|
| ベクトル検索（ChromaDB） | 意味的類似度・表記ゆれ対応 | 固有名詞の完全一致が弱い |
| BM25 | 固有名詞・専門用語 | 表記ゆれに弱い |
| RRF 統合 | 両方の長所を享受 | — |

また**親子チャンク**設計を採用した。小さな子チャンクで高精度に検索し、ヒットした箇所の親チャンク（より広い文脈）を LLM に渡す。「Sorani」のような固有名詞でも確実にヒットするようになった。

### (b) 3 層検証アーキテクチャ（ハルシネーション対策）

開発過程で **2 種類のハルシネーション**を発見・分類した。

1. **変数展開漏れ** — `{product_name}` 等のプレースホルダがそのまま出力される
2. **LLM 自己生成** — 存在しない URL や、リードが関心を示していない製品への言及

それぞれを別レイヤーで対処する設計に到達した。

| Layer | 手法 | 検出対象 |
|---|---|---|
| Layer 1 | Python 機械チェック（4 種のルール） | 内部 ID 漏れ・example.com・未設定 URL・関心外製品の言及 |
| Layer 2 | LLM-as-a-Judge（任意有効化、コスト管理可能） | 論理矛盾・文脈逸脱 |
| Layer 3 | Gmail 下書きで送信前の人間確認 | 最終確認 |

「ハルシネーション率 N% 削減」のような比較数値は測定していないが、**検出できる種類を明確に定義・実装したこと**が品質改善の基盤になった。

### (c) Claude Code Skills 活用（v1 → v2 ピボット）

**v1**: Streamlit ベース UI を実装したが、UI の保守・改善コストが高く、本質機能（RAG 精度・ハルシネーション対策）への投資時間が確保できなくなった。

**v2（現在）**: Claude Code Skills（Markdown による対話定義）に全面ピボット。UI コードをすべて削除し、操作フローは Markdown 1 ファイルで定義する方式に切り替えた。

動くものを捨てる判断は心理的コストが高かったが、「ユーザーインターフェースは Claude Code が担う」と割り切ったことで保守性が大幅に向上した。

## 結果と学び

| 項目 | 測定値 |
|---|---|
| 自動テスト | 117 件パス（開発初期の 52 件から段階的に拡充） |
| 実 LeadManager CSV カラム自動マッピング | 13 / 15 = 87% |
| エンドツーエンド処理（15 件・A〜C ランク） | 約 8 分・エラー 0 件（KB 構築 2 分 + メール生成 6 分、クリーン環境で実測） |
| 製品 RAG ヒット率 | 処理した全リードで関連資料 2〜3 件を取得 |
| 必須フィールドマッピング成功率 | 開発中盤 10/11 → 最終的に 11/11 |

**学んだこと**:

- **MVP 順序の重要性** — 「動くもの → 品質改善 → 検証強化」の順で進めたことで、本当に必要な改善が見えてきた
- **ハルシネーション対策は「層」で考える** — 単一の対策より、検出 × 訂正 × 人間確認の多層構造が堅牢
- **データ駆動設計** — コード分岐より「ランク別プロンプトテンプレート」の方が保守性が高い
- **Atomic 原則の重要性** — ナレッジベース再構築時に既存データを消去するバグを発見・修正。副作用のない設計の重要性を体感した

## 技術スタック

| 分類 | 技術 |
|---|---|
| LLM | gpt-4.1-nano (OpenAI) |
| Embedding | text-embedding-3-small |
| RAG | ChromaDB + BM25 ハイブリッド + 親子チャンク |
| 音声文字起こし | Whisper |
| Gmail | Gmail API（OAuth 2.0 / `gmail.compose` スコープ） |
| UI | Claude Code Skills (Markdown) |

## セットアップ

```bash
# 1. 依存パッケージをインストール
pip install -r requirements.txt

# 2. .env を作成して OPENAI_API_KEY を設定
echo "OPENAI_API_KEY=sk-..." > .env

# 3. Claude Code で /setup を実行
/setup
```

### 主な Skill

| Skill | 機能 |
|---|---|
| `/setup` | 初回セットアップガイド（9 ステップ） |
| `/email-workflow` | メール生成 → Gmail 下書き保存 |
| `/inspect-data` | リード CSV データ品質チェック |
